## 二、核心数据流

```
摄像头帧 (30fps)
    │
    ▼  每 250ms 采样一次
captureFrame() → dataURL (JPEG base64, quality=0.8)
    │
    ▼  WebSocket 发送
{ type: "frame", data: { image: "<base64>", timestamp: ... } }
    │
    ▼  后端 WebSocket 端点
1. base64 解码 → cv2.imdecode → BGR numpy (H,W,3)
2. RealtimeCTCService.add_frame(frame)
    │
    ├── 缓冲区 < 4 帧 → 返回 None（继续累积）
    │
    └── 缓冲区 >= 4 帧 → 触发推理：
        │
        ├── ResNet18 逐帧提取特征 → (T, 512)
        ├── BiLSTM + CTC 前向传播 → log_probs (T, 1, C)
        ├── CTC Greedy 解码 → tokens + sentence
        ├── Softmax 置信度 = mean(token_probs) × 100
        └── 查表补充 pinyin + meaning
            │
            ▼  WebSocket 返回
{ type: "result", data: { text, pinyin, meaning, confidence } }
    │
    ▼  前端 useWebSocket.onmessage
recognitionStore.setResult() → 更新 UI 面板
```

### 4.2 实时推理服务：`ctc_service.py — RealtimeCTCService`

**文件位置：** `Project/Back/app/ctc_service.py`（第 180-346 行）

**设计：滑动窗口 + 定时批量推理**

```
参数:
  max_buffer_frames = 64      # 缓冲区最大帧数（滑动窗口上限）
  infer_every_n_frames = 4   # 每累积 4 帧触发一次推理

帧缓冲区示意:
  [f1, f2, f3, f4, f5, f6, f7, f8, ... f64]
   ←── 推理窗口 ──→
                      ←── 推理窗口 ──→
```

**`add_frame(img_bgr)` 方法：**

```python
def add_frame(self, img_bgr: np.ndarray) -> dict | None:
    self.frame_buffer.append(img_bgr)
    self.frames_since_last_infer += 1

    # 滑动窗口截断
    if len(self.frame_buffer) > self.max_buffer_frames:
        self.frame_buffer = self.frame_buffer[-self.max_buffer_frames:]

    # 触发条件：累积达 4 帧
    if self.frames_since_last_infer >= self.infer_every_n_frames:
        self.frames_since_last_infer = 0
        return self._infer()

    return None  # 不满足条件，继续累积
```

**`_infer()` 方法（单次前向传播）：**

```python
def _infer(self) -> dict:
    # 1. ResNet18 特征提取（逐帧批量送入 GPU）
    features = self.extractor.extract(self.frame_buffer)  # (T, 512)

    # 2. BiLSTM + CTC 前向传播
    feats_t = torch.from_numpy(features).float().unsqueeze(0).to(device)
    log_probs, _ = self.model(feats_t, feat_lens_t)  # (T, 1, C)

    # 3. CTC Greedy 解码（折叠重复 + 去 blank）
    best_path = log_probs.argmax(dim=-1)  # 各时间步取最大概率
    collapsed = fold_consecutive(best_path)  # 去重
    tokens = [vocab.id2token[i] for i in collapsed if i != BLANK_ID]
    sentence = "".join(tokens)

    # 4. Softmax 置信度
    probs = torch.softmax(log_probs, dim=-1)
    token_probs = [probs[t, idx] for each predicted token]
    confidence = mean(token_probs) * 100  # 百分比

    return { "text": sentence, "tokens": tokens, "confidence": confidence }
```

**模型加载（`_load_model`）：**
- 调用 `demo_infer.load_checkpoint()` 加载 `Project/Back/model/last_checkpoint.pt`
- 调用 `demo_infer.build_feature_extractor()` 构建 ResNet18 特征提取器
- **全局单例**：`get_realtime_ctc_service()` 确保模型只加载一次

### 4.3 拼音/释义查表

**位置：** `main.py` 第 207-230 行

从 `DICTIONARY_DATA`（100 条手语词条）构建 `_TEXT_LOOKUP` 字典：

```python
_TEXT_LOOKUP = {
    "你好":  {"pinyin": "nǐ hǎo",  "meaning": "表示问候或打招呼"},
    "谢谢":  {"pinyin": "xiè xiè", "meaning": "表示感激"},
    # ... 共 100 条
}

def lookup_text_info(text: str) -> dict:
    # 1. 精确匹配
    if text in _TEXT_LOOKUP: return _TEXT_LOOKUP[text]
    # 2. 反向包含（文本包含某词条 → 返回该词条信息）
    for kw, info in _TEXT_LOOKUP.items():
        if kw in text: return info
    # 3. 未匹配
    return {"pinyin": "", "meaning": ""}
```

### 4.4 CTC 模型架构

**模型定义：** `AAA/ctc_model.py — TemporalConvBiLSTM`

```
输入: (B, T, 512) — 变长特征序列
  │
  ▼
1D-CNN (kernel=3, stride=1, pad=1)
  → BatchNorm1d → ReLU → Dropout
  → 保持时间长度 T 不变
  │
  ▼
BiLSTM (hidden=256, layers=2, bidirectional)
  → pack_padded_sequence 高效处理变长
  │
  ▼
Linear(512 → num_classes)
  → LogSoftmax
  │
  ▼
输出: (T, B, num_classes) — CTC 格式
```

**特征提取器：** `AAA/extract_features.py — ResNetFeatureExtractor`
- ResNet18（ImageNet 预训练）去掉 FC 层
- 自适应平均池化 → 512 维特征向量
- 逐帧处理，batch_size=16 防止显存溢出

## 七、关键设计决策

1. **滑动窗口而非逐帧推理**  
   单帧对 CTC 模型意义不大（CTC 依赖时序上下文）。累积多帧后批量推理，既保证准确率又控制开销。

2. **后端单例模型加载**  
   `get_realtime_ctc_service()` 全局单例，避免每个 WebSocket 连接重复加载模型（ResNet18 + BiLSTM 总计约 50MB）。

3. **单次前向传播**  
   旧版 HTTP 接口存在双重推理（先 `predict_from_features` 获取文本，再手工调用模型获取置信度）。新版合并为一次前向，从同一组 `log_probs` 同时完成解码和置信度计算。

4. **前端置信度由后端提供**  
   旧版前端用 `getRandomConfidence()` 生成 92%-98% 的随机值。新版置信度是模型 softmax 输出的真实概率均值，具备实际参考意义。

5. **base64 而非二进制**  
   选择 base64 而非 WebSocket binary frame，因为 JSON 协议更易调试、兼容性更好，且每帧约 30-50KB 的 base64 开销在可控范围内。

6. **拼音/释义异步查表**  
   CTC 模型仅输出字符序列（如"你好"），不包含发音和解释。通过服务端词典查表补充，前端无需额外请求。

## 附录：旧版本简介（v1.0 — HTTP 轮询）

> 此版本已废弃，保留说明供参考。

### 架构

```
摄像头 → captureFrame() → base64 → dataUrlToFile() → FormData
  → $fetch POST /api/v1/recognize/upload (每 1 秒)
  → 后端写临时文件 → ctc_service.recognize_image_file()
  → 每次请求重新加载模型 → 推理 → JSON 响应
  → 前端 getRandomConfidence() 覆盖置信度 → 展示
```

### 与新版主要差异

| 维度 | v1.0 (HTTP 轮询) | v2.0 (WebSocket) |
|------|-----------------|-----------------|
| 通信协议 | HTTP POST | WebSocket |
| 帧率 | 1 FPS (1000ms) | ~4 FPS (250ms) |
| 模型加载 | 每次请求重新加载 | 全局单例（仅一次） |
| 置信度 | 前端随机 92-98% | 后端 Softmax 真实值 |
| pinyin/meaning | 始终为空字符串 | 词典查表补充 |
| 帧编码 | base64 → File → FormData | 直接 base64 字符串 |
| 推理策略 | 单帧推理（无时序上下文） | 滑动窗口多帧推理 |
| 请求堆积控制 | `isRequesting` 互斥锁 | 无需（WebSocket 全双工） |
| WebSocket 模块 | 已编写但未接入 | 核心通信方式 |
| 骨架叠加层 | 空 canvas | 同上（仍预留） |

### v1.0 已知问题（已在 v2.0 修复）

1. **HTTP 轮询而非实时** — 每秒一次 HTTP POST，延迟高、带宽浪费
2. **置信度假数据** — `getRandomConfidence()` 返回随机值，完全不可信
3. **拼音/释义为空** — 后端返回空字符串，UI 展示空白
4. **每次请求重加载模型** — `predict_from_image()` 内部调用 `load_checkpoint()`，严重浪费资源
5. **WebSocket 模块闲置** — `useWebSocket.ts` 已实现完整逻辑但未被页面引用
6. **单帧推理无时序上下文** — CTC 模型设计依赖多帧序列，单帧效果极差
